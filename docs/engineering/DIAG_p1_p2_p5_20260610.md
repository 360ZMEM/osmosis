## 1. 诊断入口

### P1：部署期 STDW gating

原设想：episode 起始 200 step STDW off，若 `fmse_short < 0.10` 则关闭 STDW，否则开启。

本轮离线复查发现：直接使用前 200 step 不能可靠分类，因为所有 embodiment 在 150-200 step 已经足够低，阈值会把 `base/long_body` 也误关。

| Embodiment | warm0-200 | warm100-200 | warm150-200 | final200 |
|---|---:|---:|---:|---:|
| base | 0.1476 | 0.0639 | 0.0294 | 0.0284 |
| long_body | 0.1437 | 0.0666 | 0.0305 | 0.0295 |
| heavy_moderate | 0.0969 | 0.0164 | 0.0018 | 0.0282 |
| asymmetric | 0.1222 | 0.0373 | 0.0138 | 0.0259 |

结论：P1 不能只用 `fmse_short < 0.10`。更可靠的 gate 需要引入 `com_to_cob` 静态读数或微扰响应。

### P2：asymmetric 方向性 drift

关键观察：`asymmetric` 初始 `com_to_cob = (0.05, 0.05, 0.01)`，而默认 STDW `--drift_axes 0 --target_drift 0.05` 会把 x 从 `0.05` 推到 `0.10`。这不是修偏移，而是在已有偏移上继续加偏移。

### P5：yaw stage2 副作用联动

初始报告怀疑 `yaw MSE +55%` 与 `asymmetric + STDW on` 同源。本轮通道分解显示，48-cell 中 asymmetric 劣化主要来自 **pitch + depth**，yaw 不是主因。

## 4. 机理修正

旧假设：
- A3 已学到 asymmetric，STDW drift target 变成纯扰动。

新假设：
- 更精确地说，`asymmetric` 的初始偏移已经是 `(x,y)=(0.05,0.05)`；
- 当前 STDW wrapper 的 drift 语义是 `base_offset + frac * target_drift`，不是“向目标偏移收敛”；
- 因此默认 `target_drift=+0.05, axes=0` 会把 asymmetric 推到 `(0.10,0.05)`；
- pitch/depth 耦合被强激发，造成 +158%；
- 若把 drift 改成反向修正 `(-0.05,-0.05)`，结果接近 base。

## 5. 可回退实现建议

### P1 修正：从 `fmse_short gate` 改成 `offset-aware drift router`

不要直接按 `fmse_short < 0.10` 关 STDW。建议部署期 gate 改为：

```text
read base com_to_cob xy
if ||xy|| < 0.02:
    use STDW with +x drift target (base/long_body/heavy_moderate)
elif ||xy|| >= 0.04:
    use STDW with corrective drift target = -xy on axes where |offset| is large
else:
    run micro-probe (+0.01 / -0.01) and choose lower short-horizon error
```

可回退实现方式：
- 新增 CLI：`--auto_drift_router True/False`，默认 False；
- 新增 CLI：`--drift_router_mode off|offset_correct|micro_probe`，默认 off；
- 不改变 `--target_drift` / `--drift_axes` 现有语义；
- 所有新行为只在显式打开时生效。

### P2 修正：asymmetric 鲁棒化不应先重训，先做方向路由

短期：
- 对 asymmetric 使用 `target_drift=-0.05 --drift_axes 0,1`；
- 或自动读取初始 offset，令 `target = -offset_xy`。

中期：
- 训练时把 `com_to_cob_offset_xyz_range` 从 stage2 默认 `[0.03,0.03,0.01]` 扩到 `[0.05,0.05,0.01]`；
- 同时给 policy/slow-loop 显式输入 `com_to_cob` 或 offset proxy，否则 a_gain head 只能从姿态误差间接推断。

### P5 修正：yaw 不是 asymmetric 的主因，但仍需专项

本轮显示 asymmetric 劣化主因是 pitch/depth。P5 应拆成两个分支：
- P5a：stage2 yaw MSE +55% 的原始问题，继续用 stage2 play 轨迹做 yaw gain/ripple 专项；
- P5b：asymmetric + drift 的 pitch/depth 耦合问题，纳入 P2 router / domain randomization。

## 7. P5a：stage2 yaw 副作用专项（第一轮）

### 7.1 原始问题复核

7.6 节记录了 A3 stage1 vs A3 stage2+STDW：

| 指标 | A3 stage1 | A3 stage2+STDW | Δ |
|---|---:|---:|---:|
| yaw MSE | 7.85e-4 | 1.22e-3 | +55% |
| yaw ripple_std | 0.0088 | 0.0241 | +174% |
| total MSE | 9.84e-2 | 8.79e-2 | -11% |

本轮重新检查原始命令后发现一个混杂因素：
- stage1 原始 play：`batch_size=256`，slow-loop 触发 7 次；
- stage2 原始 play：`batch_size=64`，slow-loop 触发 10 次；
- 因此原始“stage2 导致 yaw +55%”并不严谨，需要把 ckpt 与 online slow-loop/batch 解耦。

### 7.5 P5 当前结论

1. 原始“stage2 yaw +55%”更准确应表述为：`batch=64 + slow_loop_interval=60` 的 online STDW 在中段窗口放大 yaw。
2. `stage2 ckpt` 本身并不导致 yaw 劣化；在 `use_stdw=False` 下，stage1/stage2 yaw 基本一致。
3. `batch=256` 能抑制 yaw，但削弱 pitch/depth 收益，说明问题不是“是否更新”，而是“更新过密/过早”。

5. `lambda_reg`、`pseudo_decay`、`pseudo_gate_limit` 在这组 720-step 探针中几乎无影响，暂不作为 P5 主线。

### 9.7 probe 评分改进：A/B/A baseline bracket

问题复盘：

- 9.6 的 `probe_i120` 全部选择 `axis1_neg`，主要原因不是所有机型都需要同一 drift；
- 原评分把最早的 baseline window 和后续候选 window 直接比较，tracking error 会随时间自然下降，因此“后跑的候选”容易占优；
- 第一版改进加入穿插 baseline 后仍使用 pooled baseline mean，仍未完全消除时间趋势。

已实现的改进：

```bash
--micro_probe_score_mode paired_axis
--micro_probe_baseline_each_candidate True
--micro_probe_min_improvement_abs 0.01
--micro_probe_min_improvement_rel 0.03
--micro_probe_consistency_margin_abs 0.005
--micro_probe_consistency_margin_rel 0.03
```

实现逻辑：

- `legacy_mean`：保留旧行为，最低均值胜出，用于复现 9.6；
- `relative_improvement`：候选必须优于其 baseline；
- `paired_axis`：候选必须优于本地 baseline，并且同轴正/负方向的 improvement 差距超过 consistency margin；
- `baseline_each_candidate=True` 时，probe schedule 变为 A/B/A bracket：每个候选只和前后两个 zero-drift baseline window 的均值比较；
- 若改善不足或正/负方向不一致，则选择 `baseline`，即 `target=0.0, axes=[]`，最小扰动优先。

新增 summary 字段：

- `micro_probe_score_mode`
- `micro_probe_score_details`
- `micro_probe_selected_improvement`
- `micro_probe_selected_reason`
- `micro_probe_baseline_each_candidate`
